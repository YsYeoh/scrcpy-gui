# scrcpy-gui — Screen recording to file (session capture)

**Date:** 2026-04-27  
**Status:** ready for implementation planning  
**Depends on:** `2026-04-28-scrcpy-gui-ux-and-wireless-design.md` (baseline app behavior; this spec supersedes its deferral of **recording** for the features listed below)  
**Pinned tooling:** [scrcpy 3.3.4 for Windows](https://github.com/Genymobile/scrcpy) as in `src/scrcpy_gui/data/vendor-windows.json` and `mirroring_options.py` (“vendored scrcpy 3.3.x”)

---

## 1. Purpose and scope

**Goal:** Let users **record the current mirroring session** to a file on the PC by passing scrcpy’s `--record=PATH` (short `-r` is equivalent; implementation may use long form in code for clarity), without exposing the full scrcpy option surface.

**In scope**

| Item | Description |
|------|-------------|
| **Record to file** | When enabled, the single managed scrcpy `QProcess` is started with **`--record=<absolute path>`** in addition to existing `mirroring_options` extra args. |
| **Output path policy** | **Remember** the last **output directory** in `QSettings`. **Autogenerate** filenames of the form `scrcpy-record-YYYYMMDD-HHMMSS.mp4` in **local** time for normal starts. **No** save dialog on Start unless the user has chosen **Save as…**. |
| **Save as…** | **Optional** action: user picks a **full file path**; that path is used for the **next** Start that has recording enabled, **once** (in-memory one-shot; not persisted). After that start, the app clears one-shot; **parent directory** of the chosen file becomes the stored **last output directory**. |
| **Audio in file** | **Match the mirror:** the recorded file’s streams follow the same capture flags as the session. Do **not** add a **recording-only** `--no-audio` (or similar) in v1. If a future change adds `--no-audio` (or other stream options) to mirroring, the same `argv` applies to a session that also uses `--record`. (Today’s `build_scrcpy_args` does not set `--no-audio`.) |
| **UI** | A **“Record to file”** checkbox (off by default), and **“Save as…”** enabled when recording is on; place controls in or directly under the existing **Mirroring quality** group so they stay discoverable with other session options. |
| **Support / Copy details** | Extend the copied support text so it notes **whether recording was enabled** for the current process (or last launch context as feasible) and the **resolved** output path scrcpy was given, consistent with the existing `mirroring extra args` line. |
| **Platform** | **Windows 10/11** only, matching the product. Use paths and `QSettings` the same way as the rest of the app. |

**Out of scope (v1)**

- **Headless** record-only sessions (`--no-playback`, `--no-window`, etc.).  
- **`--time-limit`**, `--record-orientation`**, and other scrcpy recording extras not required for basic “mirror + file”.  
- **Free-form custom scrcpy args** (still deferred; see 2026-04-28 UX spec).  
- **Pause / resume** recording in the GUI; **segmented** recordings. Stopping the scrcpy process (or the device path ending the session) defines when the file ends, per upstream.  
- **Per-device** or **per-serial** output folders.  
- Non-MP4 as the **default** autogenerated container: v1 uses **`.mp4`** for autogenerated names; **Save as…** may choose another **supported** extension (e.g. **`.mkv`**) per [upstream recording docs](https://raw.githubusercontent.com/Genymobile/scrcpy/master/doc/recording.md).

## 2. Architectural approaches and recommendation

### Option 1 — Inline in `MainWindow`

All path logic, one-shot state, and `--record` assembly live in `main_window.py` alongside existing mirroring state.

- **Pros:** Fewest new files.  
- **Cons:** The file is already large; path rules and collision handling are hard to unit test.

### Option 2 — Small pure module (recommended)

Add a focused module (e.g. `recording_paths.py` or `recording.py`) with:

- **Default directory** for first run: `%USERPROFILE%\Videos` when that path exists; otherwise the user’s home directory (or another single documented fallback—implementation picks one and tests document it).  
- **`next_automatic_path(last_dir: Path) -> Path`** with the fixed name pattern.  
- **`build_record_arg(resolved: Path) -> list[str]`** returning `["--record=" + str(resolved)]` using an **absolute** path string suitable for `argv` (no `shell=True`).  
- **Helpers** to validate that `last_dir` is a directory and to optionally create the directory for a planned file if policy requires it (see §4).

`MainWindow` (or a tiny UI helper) owns: `QFileDialog` for **Save as…**, `QSettings` get/set, and an **in-memory** `Path | None` for one-shot.

- **Pros:** Testable, matches the “pure” style of `mirroring_options`.  
- **Cons:** One additional module.

### Option 3 — Stateful controller class

- **Cons:** Unnecessary for v1 (YAGNI).

**Recommendation:** **Option 2**.

## 3. Data flow and settings

1. On **load**, read `mirroring/record_enabled` and `mirroring/record_last_dir` (see §5). If `record_last_dir` is missing or invalid, use the **default directory** (§2) without overwriting settings until a successful user action or a successful **Start** resolves a new directory.  
2. User checks **Record to file** and (optionally) **Save as…** to set a one-shot **full path** for the next Start.  
3. On **Start mirroring** with recording enabled, compute the **final path**:  
   - If one-shot is set, use it, then **clear** one-shot; set `record_last_dir` to `one_shot.parent`.  
   - Else, use `Path(record_last_dir) / <autogenerated name>.mp4`.  
4. If the **target file’s parent** does not exist, **try** `Path.mkdir(parents=True, exist_ok=True)` on the parent. If that raises **OSError**, **log** a clear one-line error and **do not** start scrcpy.  
5. **Concatenate** args: existing `build_scrcpy_args(...)` **first**, then the record fragment (append `--record=...` to the end of the extra-args list so logs match “quality toggles, then output”). If integration testing shows a scrcpy ordering bug, the implementation may reorder with a one-line note in the plan.  
6. **QProcess** starts as today; **no** second process.

---

## 4. Error handling and edge cases

- **Invalid or stale `record_last_dir`:** Fall back to default base directory; **log** a single informative line.  
- **scrcpy / write failures:** Rely on existing scrcpy stdout in the log and `QProcess` `error` / exit handling; do not add a new class of blocking dialogs unless the project already does for “failed to start scrcpy.”  
- **Filename length / reserved characters on Windows:** Sanitize is **not** required in v1 if the pattern is fixed ASCII; if the user’s **Save as** path is invalid, Qt dialog and/or start failure + log is sufficient.  
- **Concurrent sessions:** Still only **one** scrcpy instance; no change.

## 5. `QSettings` keys (authoritative for implementation)

| Key | Type | Description |
|-----|------|-------------|
| `mirroring/record_enabled` | string boolean style (`"true"`/`"false"`) | Same pattern as `mirroring/stay_awake`. |
| `mirroring/record_last_dir` | string | Absolute path of the directory used for autogenerated filenames. |

**Not persisted:** one-shot “Save as” full path (memory only, cleared after use on Start).

## 6. Testing

- **Unit tests** (preferred): default directory selection edge cases, autogenerated filename format, one-shot “use once” behavior, and that `build_record_arg` (or equivalent) returns the expected `--record=...` fragment for a `Path` input.  
- **Manual:** Enable recording, Start, verify file appears and is playable; **Stop** and verify file is closed; use **Save as** then Start twice and verify second run uses a new autoname in the same folder unless a new one-shot is set.
